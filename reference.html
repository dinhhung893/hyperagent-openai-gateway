OpenAI-compatible backend

A gateway that speaks the OpenAI wire protocol to any compatible client and translates every call into Hyperagent operations — exposing the complete OpenAI attribute surface and Hyperagent's entire tool catalog (Shell, Write, web, browser, media, tables, docs, maps, integrations) through tool_calls.

OpenAI client ──▶ Gateway ──▶ MCP JSON-RPC ──▶ hyperagent.com/api/mcp ──▶ agent run

The upstream · confirmed 2026-07-19

Hyperagent exposes exactly one programmatic surface: the MCP server

Confirmed by docs + live endpoint probing. This is the whole contract the gateway builds on — there is no REST/GraphQL/OpenAPI/SDK alternative.

Transport

JSON-RPC 2.0 over POST

Streamable HTTP at hyperagent.com/api/mcp. GET → 405. Per-response SSE only; no long-lived stream channel.

Auth

OAuth 2.1 · no API keys

Authorization Code + PKCE (S256), Dynamic Client Registration, refresh via offline_access. Browser sign-in once per user; token acts as that user.

Execution

Async · poll to collect

create_thread returns a threadId immediately; poll get_thread until not running. No token-level streaming → the gateway emulates it.

The bridge · Shell · Write · tool_calls

How the whole tool catalog reaches OpenAI clients

The MCP surface can't invoke agent tools directly, so the gateway bridges them in three modes — all using the standard OpenAI function-calling format, so unmodified clients just work.

Mode A · Observability

See what the agent did

Tool activity in the thread (Shell, Write, web search, image gen…) is surfaced back as OpenAI tool_calls + results and Responses step events.

Mode B · Directed

Force a capability

Clients pass a canonical tool + tool_choice to steer the agent toward a specific capability.

Mode C · Tool-runner

Direct execution

Standard tool round-trip (or auto-exec) backed by a tool-runner agent → direct Shell/Write/etc. via the OpenAI protocol.

Honest by design

Known limits & how the gateway handles them

  • Latency: every call runs a full agent pipeline — slower than raw inference; background + polling.
  • Streaming is emulated: Hyperagent doesn't push partial tokens; deltas come from poll diffs.
  • Sampling params: temperature/top_p/seed are best-effort or no-op.
  • No embeddings via MCP: /v1/embeddings is a documented 501 or optional skill-backed adapter.
  • Per-user OAuth: no headless API key upstream — each user authorizes once; gateway rotates refresh tokens.
  • Exact MCP schemas: captured at build time via a live tools/list with a real token.
Naming caveat. Tool/parameter names come from public docs (product language). Verbatim MCP input schemas and the agent runtime signatures are validated against the live server before the gateway's schemas are frozen.
